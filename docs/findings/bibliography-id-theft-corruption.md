# Bibliography ID Theft Corruption

**Symptom:** Typing near the bibliography boundary (pressing Enter to create a new paragraph) corrupts bibliography entries in the database. The bibliography heading or entries lose their content and are replaced with empty text.

**Investigation duration:** 3 rounds of debugging across multiple sessions.

---

## Root Cause

`assignBlockIds()` in `block-id-plugin.ts` processes blocks in **document order** and uses **greedy proximity matching**. When the user creates a new paragraph near the bibliography boundary (e.g., pressing Enter), the new empty paragraph at pos 30 steals the bibliography entry's ID from pos 44 (same type `paragraph`, distance=14). The real bibliography entry at pos 46 then gets a temp ID. `detectChanges()` sees the bibliography ID now pointing at empty content and reports `UPDATE id=X [text,size:88→2]`, which overwrites the bibliography entry in the database.

**Diagnostic evidence (Round 2 logs):**
```
existingIds=["0:AD882656","13:29E98ACF","30:9142D244","44:2FF95936"]
newIds=    ["0:AD882656","13:29E98ACF","30:2FF95936","32:9142D244","46:temp-cf6"]
                                            ^^^^^^^^^ STOLEN from bib entry
```

The new paragraph at pos 30 was closer to the bibliography entry's old position (pos 44, distance=14) than the real bibliography entry at pos 46 (distance=2). Document-order processing meant the new paragraph claimed the ID first.

## Fix (Round 3)

### Primary Fix: Two-Phase Proximity Matching (TypeScript)

Refactored `assignBlockIds()` to use two phases when `structureChanged` is true:

**Phase 1 — Exact-position matches:** Process blocks in document order. If the block's position has an existing ID with matching type, claim it immediately. Unchanged from original.

**Phase 2 — Closest-first global matching:** Blocks that failed Phase 1 are collected into a deferred list. Instead of each block greedily grabbing the closest same-type match in document order:
1. Collect ALL (deferredBlock, candidate) pairs with distances
2. Sort all pairs by distance ascending, tiebreak by oldPos
3. Assign greedily from the sorted list — closest pairs first

This ensures the bibliography entry at new pos 46 claims its match at old pos 44 (distance=2) before the new paragraph at pos 30 can steal it (distance=14).

When `structureChanged` is false, the original per-block proximity algorithm is used (normal typing doesn't change block count).

### Safety Net: Reject Bibliography Block Updates (Swift)

Added a guard in `Database+Blocks.applyBlockChangesFromEditor` at the top of the updates loop: if the block being updated has `isBibliography == true`, skip the update. Bibliography content is machine-generated by `BibliographySyncService` and should never be modified through the editor polling pipeline. This is defense-in-depth — even if the TypeScript fix has a gap, bibliography data is protected.

## Blind Alleys

### Round 1: Single-block bibliography model

Initially, `BibliographySyncService` stored the entire bibliography (heading + all entries) as a **single block** in the database. This created a 1:N mismatch with ProseMirror, which parses it into multiple top-level nodes (one heading + N paragraphs). The block ID plugin had no way to map one database ID to multiple PM nodes.

**What we tried:** Fixing the parity mismatch by treating the bibliography as a single conceptual entity. This failed because the block-sync plugin operates at the PM node level — each node needs its own ID.

**What worked:** Split bibliography storage into one block per entry (heading block + N paragraph blocks, all with `isBibliography=true`). This gave 1:1 parity between DB blocks and PM nodes.

### Round 1: Missing empty-fragment filtering

`assembleMarkdown()` included blocks with empty `markdownFragment` values (e.g., section breaks with no content). These produced no PM nodes, creating a count mismatch between the block ID array and PM's node count. The PARITY MISMATCH warning fired, and block IDs were misaligned from that point forward.

**Fix:** Added `isEmptyFragment()` filtering in `assembleMarkdown()`, `idsForProseMirrorAlignment()`, and `firstBibliographyNodeIndex()` — all three must stay in sync.

### Round 1: Insert-before-update ordering

`applyBlockChangesFromEditor()` processed updates before inserts. When a block was both inserted (with temp ID) and immediately updated (with the same temp ID), the update ran first, failed to find the block, and was silently dropped. The insert then ran, but the update's newer content was lost.

**Fix:** Reordered to process inserts BEFORE updates. Added `idMapping` lookup so updates can find blocks that were just inserted with a different permanent ID.

### Round 2: Cursor clamping at bibliography boundary

After fixing the 1:N parity issue, a new problem appeared: the cursor could be positioned inside bibliography paragraphs, making them editable. User keystrokes in bibliography text created block-sync UPDATEs that overwrote the machine-generated content.

**Fix:** Added `cursorBoundary` parameter to `setContentWithBlockIds()`. `firstBibliographyNodeIndex()` computes the PM node index of the first bibliography block. The JS side clamps `safeFrom` to stay before `bibPos`, preventing cursor placement in bibliography paragraphs.

### Round 2: Race between bibliography notification and pending JS changes

The `.bibliographySectionChanged` notification fires after `BibliographySyncService` writes new bibliography blocks to the DB. The handler reads all blocks from DB and pushes them to the editor. But the editor may have **unsaved JS changes** (pending in the block-sync debounce timer) that haven't been flushed to DB yet. The handler reads stale DB state, and when those JS changes eventually flush, they create ghost blocks.

**Fix:** Added `pollBlockChangesNow()` — a force-mode poll that bypasses `contentState` and generation guards. Called before `fetchBlocksWithIds()` in bibliography, notes, and footnote rebuild handlers. Also added pending-notification queuing: if `.bibliographySectionChanged` or `.notesSectionChanged` arrives while `contentState != .idle`, the notification is queued and re-posted when idle.

### Round 2: Extensive diagnostic logging (NH1-NH5)

Added five diagnostic instrumentation points to trace the exact sequence of events:
- NH1: Log ALL top-level nodes (including non-block-types) in `setBlockIdsForTopLevel`
- NH2: Log the first doc-changing transaction after content load
- NH3: Stack trace on `setContent()` calls
- NH4: Log transaction step ranges in `blockSyncPlugin.apply()`
- NH5: Track intermediate transactions between content dispatch and RAF callback

These were essential for confirming the root cause (greedy proximity matching) and ruling out other hypotheses (normalization transactions, async timing). **Removed in Round 3 after fix was confirmed.**

### Round 2: Type-aware matching (necessary but insufficient)

Added `currentBlockTypes` tracking so proximity matching only considers same-type candidates. This prevented cross-type theft (e.g., paragraph stealing a heading's ID) but didn't prevent same-type theft (new paragraph stealing bibliography paragraph's ID). The two-phase algorithm in Round 3 was the actual fix for same-type theft.

## Key Lessons

1. **Greedy document-order matching fails near structural boundaries.** When a new block appears between existing blocks of the same type, document-order processing lets the new block claim a nearby ID before the rightful owner. Global closest-first matching prevents this.

2. **Machine-generated content needs write protection.** Bibliography entries should never be modified by the editor polling pipeline. A simple `isBibliography` guard prevents an entire class of corruption bugs regardless of ID matching quality.

3. **Force-flush JS changes before reading DB.** Any handler that reads blocks from DB and pushes them to the editor must first flush pending JS changes. Otherwise the handler reads stale state, and the delayed JS flush creates duplicate/ghost blocks.

4. **1:1 parity between DB blocks and PM nodes is essential.** When the block count doesn't match PM's node count, block IDs misalign and all downstream sync logic breaks. Every path that produces block ID arrays must filter consistently (empty fragments, list collapsing).
