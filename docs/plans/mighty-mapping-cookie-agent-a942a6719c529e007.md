# Plan Review: PDF Export Citation Formatting via --citeproc

## Summary

The plan proposes switching from `zotero.lua` to pandoc's `--citeproc` for PDF exports to properly format in-text citations (e.g., `[@Smith2020]` becomes "(Smith 2020)"). DOCX/ODT exports would continue using the Lua filter.

---

## 1. CRITICAL: Duplicate Bibliography

**This is the highest-risk issue in the plan.**

The exported content already contains a bibliography section generated by `BibliographySyncService`. Here is the data flow:

1. `BibliographySyncService.performBibliographyUpdate()` creates a bibliography `Block` with `isBibliography: true` in the database (`/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Services/BibliographySyncService.swift`, lines 314-348).
2. When not zoomed, `ContentView.rebuildDocumentContent()` calls `db.fetchBlocks(projectId:)` which returns ALL blocks including bibliography blocks (`/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Models/Database+Blocks.swift`, line 80).
3. `BlockParser.assembleMarkdown(from:)` concatenates all block markdown fragments, including bibliography (`/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Services/BlockParser.swift`, line 340).
4. `SectionSyncService` saves this full markdown (with bibliography) to the `Content` table (`/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Services/SectionSyncService.swift`, line 232).
5. `ExportOperations.handleExport()` calls `DocumentManager.shared.loadContent()` which reads from the `Content` table (`/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Commands/ExportCommands.swift`, line 32).

**Result:** The markdown passed to pandoc already has a "# Bibliography" section with formatted entries. When `--citeproc` is added, pandoc will ALSO generate a bibliography at the end of the document. The exported PDF will have two bibliography sections.

**Recommendation:** The export function must strip the existing app-generated bibliography from the markdown before passing it to pandoc with `--citeproc`. This can be done by:
- Detecting and removing the `# Bibliography` (or whatever `bibliographyHeaderName` is set to) section and everything under it, OR
- Better: assembling export content from blocks while explicitly excluding `isBibliography` blocks (similar to how the QuickLook extension does it in `/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/QuickLook Extension/SQLiteReader.swift`, line 47: `WHERE isNotes = 0 AND isBibliography = 0`).

This applies to all export formats, not just PDF -- the current DOCX/ODT exports likely also have this duplication problem with the Lua filter, since `zotero.lua` also generates bibliography markers.

---

## 2. CRITICAL: --citeproc Compatibility with XeTeX PDF Pipeline

The plan proposes using `--citeproc --bibliography temp.json --csl style.csl` for PDF output.

**This will work.** Pandoc's `--citeproc` operates as a pandoc filter at the AST level, before the document is handed to the PDF engine. It resolves `[@citekey]` into formatted text and generates a bibliography `Div`. The resulting LaTeX is then compiled by XeTeX. There is no conflict between `--citeproc` and the XeTeX pipeline.

However, there is a subtle ordering concern:
- If both `--citeproc` and `--lua-filter` are specified, the order matters. But since the plan says PDF will NOT use `--lua-filter`, this is not an issue.

**Verified:** The plan correctly separates the two paths (PDF uses citeproc; DOCX/ODT uses lua filter).

---

## 3. IMPORTANT: Zotero API Interaction for CSL-JSON

The plan adds a `fetchBibliographyJSON(for:)` method that calls Zotero/BBT's JSON-RPC API to get CSL-JSON.

**Observations:**

a) `ZoteroService` already has `fetchItemsForCitekeys()` at `/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Services/ZoteroService.swift`, line 337, which calls BBT's `item.export` with "Better CSL JSON" format. This returns CSL-JSON as a string. The plan should reuse this existing method rather than adding a new JSON-RPC call.

b) `ZoteroService` also has `cslJSONForCitekeys()` (line 440) which returns cached items as a JSON string. If items are already cached (which they will be if the user has been editing with citations visible), this could avoid a network call entirely.

c) The citekey extraction logic already exists in `BibliographySyncService.extractCitekeys(from:)` at line 65. The plan should reuse this rather than re-implementing extraction.

**Recommendation:** The new `fetchBibliographyJSON(for:)` should:
1. Use `BibliographySyncService.extractCitekeys(from:)` for extraction
2. First check `ZoteroService.shared` cache via `getItems(citekeys:)`
3. Only call `fetchItemsForCitekeys()` for missing items
4. Use `cslJSONForCitekeys()` to serialize the result

This follows existing patterns and avoids duplicating logic.

## 4. IMPORTANT: Lua Filter Validation Needs Adjustment

Currently, `ExportService.export()` always validates the Lua script path (lines 130-134 of ExportService.swift):

```swift
if let luaPath = luaScriptPath {
    guard FileManager.default.fileExists(atPath: luaPath) else {
        throw ExportError.luaScriptNotFound(luaPath)
    }
}
```

And always adds it to pandoc arguments (lines 187-189):

```swift
if let luaPath = luaScriptPath {
    arguments.append(contentsOf: ["--lua-filter", luaPath])
}
```

For PDF exports with `--citeproc`, the Lua filter should NOT be added. The plan says it will "branch citation processing by format" but needs to be explicit about:
- Skipping `effectiveLuaScriptPath` resolution entirely for PDF, OR
- Conditionally skipping the Lua filter argument addition based on format

The cleanest approach: add the Lua filter arguments conditionally:

```swift
// Add Lua filter for Zotero citations (DOCX/ODT only)
if format != .pdf, let luaPath = luaScriptPath {
    arguments.append(contentsOf: ["--lua-filter", luaPath])
}
```

And similarly, skip validation of the Lua script when format is PDF.

## 5. IMPORTANT: effectiveLuaScriptPath in ExportSettings

The `effectiveLuaScriptPath` computed property in `ExportSettings` (`/Users/niyaro/Documents/Code/ff-dev/pandoc-pdf/final final/Models/ExportSettings.swift`, line 111) always resolves to the bundled Lua script or a custom one. This property is still valid for DOCX/ODT exports.

For PDF, the code just needs to not use it. No changes to `ExportSettings` itself are needed -- the branching should happen in `ExportService.export()`.

**However:** The preferences UI (ExportPreferencesPane) might show Lua filter options that confuse users since they don't apply to PDF. Consider adding a note in the UI, or this can be deferred as a non-blocking concern.

## 8. SUGGESTION: Error Handling for Missing Zotero

The plan says "warn but still export if Zotero is unavailable." The current code already does this (ExportService.swift lines 194-209): it collects warnings about Zotero status and includes them in the ExportResult.

For the PDF path with `--citeproc`, if Zotero is unavailable:
- The bibliography JSON file won't be created (or will be empty)
- Pandoc `--citeproc` with an empty `--bibliography` may still run, but citations will remain unresolved as `[@citekey]` in the PDF
- This is the same behavior as the current Lua filter path when Zotero is down

The warning messages at lines 197-208 are already appropriate. The only addition needed is: if citekeys exist but bibliography JSON could not be fetched, skip the `--citeproc`/`--bibliography`/`--csl` arguments entirely so pandoc does not error out on a missing or empty bibliography file.

## 9. SUGGESTION: Consider Also Fixing DOCX/ODT

The duplicate bibliography problem (item 1 above) likely affects DOCX/ODT exports too, since the `zotero.lua` filter generates its own bibliography section (see `Doc` function at line 2115 of zotero.lua, and `zotero_bibl_odt` at line 1786). The markdown content already contains the app-generated bibliography. When zotero.lua adds its own, there would be two.

This is not caused by the current plan, but is an existing bug worth noting.

---

## Verdict

The plan is fundamentally sound in its approach (use `--citeproc` for PDF), but has one critical gap that must be addressed before implementation:

**The app-generated bibliography section must be stripped from the exported markdown when using `--citeproc`**, otherwise the output will have duplicate bibliographies.

The recommended implementation order:
1. Add bibliography stripping to the export content pipeline (affects all formats)
2. Add `bundledCSLStylePath` and copy the CSL file, update project.yml
3. Add `fetchBibliographyJSON` (reusing existing ZoteroService methods)
4. Modify `export()` to branch by format for citation processing
5. Add temp file cleanup for bibliography JSON
6. Test with and without Zotero running
