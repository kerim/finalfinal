# Code Review: Image Caption Duplication Bug Fix (Fix C)

## Summary

This review evaluates the diagnosis and proposed fix for the image caption duplication bug described in `docs/plans/idempotent-stirring-hopper.md`, specifically Fix C -- the BlockParser change to keep caption comments together with following image lines.

---

## 1. Is the Root Cause Diagnosis Correct?

**Yes, the diagnosis is correct.** The evidence chain is sound across all layers:

**JS layer (toMarkdown serializer):** In `/Users/niyaro/Documents/Code/ff-dev/images/web/milkdown/src/image-plugin.ts`, lines 134-157, the `toMarkdown` runner emits two separate mdast nodes for a single figure:

1. An `html` node via `state.addNode('html', ...)` containing `<!-- caption: text -->` (line 140)
2. A `paragraph > image` compound node via `state.openNode('paragraph')` / `state.addNode('image', ...)` (lines 149-155)

Both are **top-level flow content** in the mdast tree. The `html` node type in mdast is a block-level node. When `remark-stringify` (via `mdast-util-to-markdown` v2.1.2, confirmed in `pnpm-lock.yaml`) serializes flow-content siblings, it inserts a blank line (`\n\n`) between them. This is standard remark-stringify behavior -- block-level siblings are always separated by blank lines.

**Swift layer (BlockParser.splitIntoRawBlocks):** In `/Users/niyaro/Documents/Code/ff-dev/images/final final/Services/BlockParser.swift`, lines 144-223, the `splitIntoRawBlocks` method splits markdown on blank lines (lines 185-201). The blank line inserted by remark-stringify between the caption comment and the image markdown causes them to be split into two separate blocks:

- Block A: `<!-- caption: text -->` -- classified as `.paragraph` (falls through to default at line 282)
- Block B: `![alt](media/file.png)` -- classified as `.image` (line 272)

**The accumulation mechanism:** On each roundtrip:
1. Editor serializes figure as `<!-- caption: text -->\n\n![alt](media/file.png)`
2. Swift splits this into two blocks, storing the caption comment as a standalone paragraph
3. `assembleMarkdown` (line 358-376) joins blocks with `\n\n`, so the caption paragraph persists
4. On reload, `remarkFigurePlugin` (line 18-64) correctly consumes the caption from the preceding html node and transforms the image paragraph into a figure node with the caption attribute set
5. But the caption comment is **also** still present as Block A in the database, which gets reassembled into the markdown that is pushed to the editor
6. Over multiple roundtrips, this means the caption text appears both as a standalone paragraph AND inside the figure node

One critical observation: the `remarkFigurePlugin` (lines 59-63) does remove the caption html nodes from the mdast tree during parsing (via the `toRemove` reverse splice). So on re-parse, the caption comment is consumed and does not appear as a separate element in the editor. But the underlying database still contains the orphaned paragraph block. The duplication manifests when blocks are reassembled and re-parsed: the orphaned paragraph block keeps being included in the assembled markdown, and while it gets consumed on the next parse, it also gets regenerated by `toMarkdown` on the next save. The real problem is that Block A (the paragraph) never gets deleted from the DB.

## 2. Evaluation of the Proposed Swift-Side Fix

### Change 1: splitIntoRawBlocks peek-ahead (Fix C, Change 1)

The proposed logic is:

```swift
// In blank-line handling, after footnote check:
let trimmedBlock = currentBlock.trimmingCharacters(in: .whitespacesAndNewlines)
if trimmedBlock.range(of: "^<!--\\s*caption:", options: .regularExpression) != nil
   && trimmedBlock.hasSuffix("-->") {
    var nextIdx = index + 1
    while nextIdx < lines.count
          && lines[nextIdx].trimmingCharacters(in: .whitespaces).isEmpty {
        nextIdx += 1
    }
    if nextIdx < lines.count
       && lines[nextIdx].trimmingCharacters(in: .whitespaces).hasPrefix("![") {
        currentBlock += line + "\n"
        continue
    }
}
```

**Assessment: The logic is correct and handles the primary case well.** The peek-ahead skips additional blank lines and looks for a `![` prefix. It absorbs the blank line into the current block, keeping caption and image together.

**Edge cases to consider:**

| Edge Case | Behavior | OK? |
|-----------|----------|-----|
| Multiple consecutive blank lines between caption and image | The `while` loop skips them all, finds `![` | Yes -- correctly handles |
| Caption at end of document (no following image) | `nextIdx >= lines.count` check fails, falls through to normal flush | Yes -- correctly handles |
| Caption followed by non-image content | `hasPrefix("![")` check fails, falls through to normal flush | Yes -- correctly handles |
| Caption with multi-line content (not possible with current format) | `hasSuffix("-->")` check ensures only single-line comments match | Yes -- correctly handles |
| Image without caption | No caption block to match, no change in behavior | Yes -- no impact |
| Caption comment nested inside a code block | Code blocks are handled earlier (line 155-160), so this branch is never reached | Yes -- safe |

**One minor concern:** The `hasSuffix("-->")` check on `trimmedBlock` is good defensive programming. However, the regex `^<!--\\s*caption:` and the suffix check together mean this only matches blocks that are exactly one HTML comment. If a user somehow typed two caption comments (unlikely), only the second would be absorbed. This is acceptable behavior.

### Change 2: detectBlockType for combined blocks (Fix C, Change 2)

The proposed logic is:

```swift
// Before existing image check:
if trimmed.hasPrefix("<!--") && trimmed.contains("caption:") {
    if trimmed.range(of: "!\\[", options: .regularExpression) != nil {
        return (.image, nil)
    }
}
```

**Assessment: This is correct and necessary.** Without this change, the combined block `<!-- caption: text -->\n\n![alt](media/file.png)` would be classified as `.paragraph` (since it starts with `<!--`, not `![`). Classifying it as `.image` ensures the image metadata extraction logic (lines 93-107 in `BlockParser.parse`) runs on it.

**One concern with the image metadata extraction:** Lines 96-106 use a regex `#"!\[([^\]]*)\]\(([^)]+)\)"#` to extract `imageSrc` and `imageAlt` from the trimmed block. Since the combined block now contains both the caption comment and the image markdown, this regex will still match correctly because it searches anywhere in the string, not just at the start. So this is safe.

**However**, the `markdownFragment` stored for this combined block will include the caption comment AND the image markdown, joined by blank lines. When `assembleMarkdown` emits this block, the caption comment will be part of the output. This is the **correct desired behavior** -- the caption stays with the image.

## 3. Should the Fix Be Done on the JS Side Instead?

**This is the key architectural question.** There are two possible approaches:

### Option A: Swift-side fix (proposed)
Keep the caption as a separate `html` mdast node in `toMarkdown`, but teach `splitIntoRawBlocks` to keep them together.

**Pros:**
- Does not change the mdast structure, which remark/milkdown expects
- The `remarkFigurePlugin` already handles the separate `html` node correctly on parse
- Follows the existing pattern of Swift-side block merging (footnote definitions already use peek-ahead)

**Cons:**
- Adds complexity to an already complex block splitter
- The root cause (two separate mdast nodes) remains -- any future consumer of the serialized markdown would face the same issue

### Option B: JS-side fix (alternative)
Change `toMarkdown` to emit the caption comment INSIDE the paragraph node that wraps the image, so they serialize as one block without a blank line separator.

This would look something like:

```typescript
toMarkdown: {
  runner: (state: any, node: ProsemirrorNode) => {
    state.openNode('paragraph');
    if (caption) {
      state.addNode('html', undefined, undefined, {
        value: `<!-- caption: ${caption} -->`,
      });
    }
    state.addNode('image', undefined, undefined, {
      url: node.attrs.src || '',
      alt: node.attrs.alt || '',
      title: null,
    });
    state.closeNode();
  },
}
```

**The problem with this approach:** In mdast, a `paragraph` node's children are **phrasing content**. An `html` node at the phrasing level becomes an inline html node, not a block html node. When serialized, remark-stringify would emit: `<!-- caption: text -->![alt](media/file.png)` all on one line, without a line break between them. This might work for the roundtrip, but it produces ugly markdown and could confuse the `remarkFigurePlugin` on re-parse, since it looks for the caption html node as a **sibling** of the paragraph node (line 39: `captionMap.has(index - 1)`), not as a child within it.

Alternatively, one could try putting both inside the paragraph as separate children, but mdast paragraph children must be phrasing content, and a standalone `html` node alongside an `image` node inside a paragraph would serialize strangely.

**Verdict: The Swift-side fix is the more pragmatic approach.** The mdast structure with separate block-level nodes is semantically correct (the caption comment IS a separate block-level element in markdown). The issue is specifically with Swift's block parser not understanding the semantic relationship between them. The proposed fix teaches it that relationship, similar to the existing footnote continuation logic.

A cleaner JS-side alternative would be to **not emit the caption comment at all** in `toMarkdown`, since the caption is already stored in the Block's `imageCaption` database column via the `updateImageMeta` message handler. The caption comment is only needed for **export** (Pandoc compatibility), and that's already handled by `Block.markdownForExport()` (lines 331-352 in Block.swift). However, this would break the CodeMirror editor's ability to show captions (the CM `image-preview-plugin.ts` reads `<!-- caption: text -->` from the preceding line to display captions). So eliminating the caption comment from `toMarkdown` would require also changing the CM caption display mechanism.

## 4. Risk Assessment: Could the BlockParser Change Affect Other Block Types?

**Risk is low.** The change is narrowly scoped:

1. **The peek-ahead only triggers** when `currentBlock` matches the specific pattern `^<!-- caption:` AND ends with `-->`. No other block type in the application uses this pattern. Other HTML comments used by the app (`<!-- ::break:: -->`, `<!-- ::auto-bibliography:: -->`, `<!-- ::zoom-notes:: -->`, `<!-- ::task:: -->`, etc.) do not start with `<!-- caption:`.

2. **The `detectBlockType` change** only fires when the block starts with `<!--` AND contains `caption:` AND contains `![`. This is a very specific combination that only matches caption+image blocks.

3. **The footnote peek-ahead at lines 186-196** is unaffected because `inFootnoteDef` is a separate state flag that gets checked first. The caption check only runs after the footnote check falls through (when `inFootnoteDef` is false or the footnote def has ended).

**One consideration:** If a user typed a paragraph that happened to start with `<!-- caption:` and end with `-->`, and it was followed by an image on the next line after a blank line, the parser would incorrectly merge them. This is extremely unlikely for real content and the pattern is specific enough that I would not consider this a practical risk.

## 5. Additional Observations

### The caption is stored in two places

The caption currently lives in:
1. The `imageCaption` column in the Block database table (set via `updateImageMeta` message handler)
2. The `markdownFragment` of the image block (when the editor serializes, the caption comment appears in the markdown)

After Fix C, the `markdownFragment` for an image block with a caption would be:
```
<!-- caption: text -->

![alt](media/file.png)
```

This means the caption is encoded in `markdownFragment` (as an HTML comment) AND in `imageCaption` (as a plain string). This redundancy is intentional -- `markdownFragment` is the source of truth for editor content, while `imageCaption` is metadata for programmatic access and export.

### The `markdownForExport()` method could produce double captions

Looking at `Block.markdownForExport()` (lines 331-352 in Block.swift), it generates the caption comment from `imageCaption` and prepends it to the image markdown. But after Fix C, the `markdownFragment` already contains the caption comment. Since `markdownForExport()` generates caption from `imageCaption` (not from `markdownFragment`), and it constructs the entire output from `imageSrc`/`imageAlt`/`imageCaption`/`imageWidth` rather than using `markdownFragment`, this is not a problem -- `markdownForExport()` ignores `markdownFragment` entirely for image blocks.

However, `assembleMarkdown()` (used for editor content) uses `markdownFragment` directly. So the caption will appear as part of the fragment. This is the correct behavior for the editor roundtrip.

### The `assembleMarkdownForExport()` path

`assembleMarkdownForExport()` (line 380-391) uses `markdownForExport()` which reconstructs image markdown from DB columns. The caption comment in `markdownForExport()` includes a `\n\n` separator (line 340: `result += "<!-- caption: \(caption) -->\n\n"`). This produces the same blank-line-separated format. If this exported markdown were ever re-imported, it would hit the same splitting issue. But since export is a one-way operation (to Pandoc), this is not a concern.

## 6. Verdict

| Question | Answer |
|----------|--------|
| Is the root cause diagnosis correct? | **Yes** -- confirmed across JS serialization and Swift block parsing layers |
| Will the proposed Swift-side fix work? | **Yes** -- the peek-ahead logic correctly handles all identified edge cases |
| Is a JS-side fix preferable? | **No** -- the mdast structure is semantically correct; the Swift parser is the right place to handle this |
| Are there risks to other block types? | **Minimal** -- the pattern match is highly specific to caption+image combinations |

### Recommendations

**No critical issues found.** The proposed Fix C is well-designed and safe to implement.

**Important (should address):**

1. After implementing Fix C, verify that existing orphaned caption paragraph blocks in the database are cleaned up. On the next content roundtrip after the fix, the caption comment will be absorbed into the image block, but any existing standalone `<!-- caption: text -->` paragraph blocks in the DB need to be handled. The natural roundtrip should handle this: on next save, the editor emits caption+image together, Swift now keeps them as one block, and the old separate caption block will no longer appear in the parsed output. However, it depends on whether the block sync uses full re-parse or incremental updates. Since `BlockSyncService.parseAndStoreBlocks` calls `replaceBlocks` (which presumably replaces all blocks), the orphaned blocks should be cleaned up on next full re-parse.

2. Consider adding a unit test for `splitIntoRawBlocks` that verifies caption+image blocks stay together. The test should cover: caption followed by image (normal case), caption followed by blank lines then image (multiple blanks), caption at end of document (no image), and caption followed by non-image content.

**Suggestions (nice to have):**

1. The long-term cleanest solution would be to not emit caption comments in `toMarkdown` at all, since captions are stored in the DB's `imageCaption` column and only needed for export (already handled by `markdownForExport()`). This would require updating the CM image preview plugin to get captions from block metadata rather than from preceding HTML comments. This is a larger refactor and not necessary for this fix.
